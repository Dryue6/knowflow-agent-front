# Codex 项目协作规范

## 中文注释要求

当编写或修改代码时，应为所有新增或变更的方法、接口、关键逻辑和重要业务逻辑补充清晰的中文注释。注释应说明代码的业务意图、关键判断依据、输入输出约束、异常处理原因或复杂流程的执行步骤，帮助后续维护者快速理解设计目的。

注释应写在最贴近相关代码的位置：方法和接口使用方法级注释，复杂分支、核心算法、状态流转、外部依赖调用、降级策略、缓存策略、并发处理和重要边界条件使用行内或块级注释。

避免添加无信息量的注释，例如仅重复变量名、方法名或语法含义的说明。已有代码含义已经足够直观时，不需要为了注释而注释；但涉及业务规则、隐含约束或容易误改的逻辑时，必须写明原因。

## 文档更新流程

在每次执行代码修改或 Bug 修复任务结束后，请严格执行以下文档更新流程：

定位文档：检查 /docs 目录下是否存在与当前修改功能相关的 Markdown 文档。

若存在，直接在文件末尾追加新记录。

若不存在，请在 /docs 目录下创建一个以功能命名的新 .md 文件（例如 /docs/feature-name-log.md）。

自检机制：在输出回答前，请自我确认：“我是否已经将刚才的改动记录到了 docs 文件夹中？”如果没有，请立即补充。

写入记录：在新建或找到的文件中，按以下模板追加本次的修改日志：

```markdown
[YYYY-MM-DD] 修改摘要
- 修改内容：[详细描述你做了哪些改动]
- 注意事项：[列出需要注意的边界情况或后续建议]
```
